# Life CFO - Financial Pressure Signals
Last updated: 2026-03-14

This document describes the current implemented pressure signal system.

---

# Purpose

Pressure signals explain why the current household money picture may feel steady, tight, or uncertain.

Signals are derived from household truth and reused by:
- snapshot output
- Money Hub explanation
- money Ask diagnosis/planning/affordability/scenario responses

---

# Current Signals

Implemented in `lib/money/reasoning/pressureSignals.ts`.

## 1) Structural Pressure

Uses:
- monthlyized recurring income
- monthlyized recurring bills

Key behavior:
- if bills exist without recurring income, pressure is treated as elevated baseline risk
- otherwise score follows committed share of recurring income

Summary examples generated by code include:
- "A large share of recurring income is already committed."
- "Commitments leave room for discretionary choices."

## 2) Discretionary Drift

Uses:
- outflow over recent 30 days
- outflow over prior 30 days

Key behavior:
- compares recent vs prior window
- marks higher pressure when recent flexible outflow rises materially

Summary examples generated by code include:
- "Flexible spending is running well above the prior period."
- "Flexible spending is at or below the prior period."

## 3) Timing Mismatch

Uses:
- available cash
- bills due before next income date
- next recurring income timing

Key behavior:
- computes shortfall before income arrives
- higher shortfall relative to obligations/cash increases pressure score

Summary examples generated by code include:
- "Key obligations land before income and exceed available cash."
- "Income timing appears adequate for upcoming obligations."

## 4) Stability Risk

Uses:
- recurring income source coverage
- bills-without-income condition
- connection freshness age

Key behavior:
- combines multiple components (income coverage, balance mismatch, freshness)
- older sync age increases risk component

Summary examples generated by code include:
- "Income reliability and data freshness look unstable."
- "Income rhythm and data freshness appear stable."

---

# Scoring and Levels

Each signal has:
- `score` (0..1)
- `level` (`none`, `low`, `medium`, `high`)
- `summary`
- `drivers[]`
- `evidence{}`

Current level thresholds:
- high: `>= 0.75`
- medium: `>= 0.5`
- low: `> 0.15`
- none: `<= 0.15`

---

# Data Quality and Freshness Effects

Signal behavior accounts for missing or stale data.

Examples:
- missing recurring setup can produce "cannot be assessed" style messaging
- stale connection age contributes to stability risk
- no setup is treated as unknown/limited certainty, not user blame

---

# How Signals Are Used

Signals currently feed:
- `buildFinancialSnapshot` (`pressure` field)
- `explainSnapshot` pressure text
- `/api/money/ask` diagnosis/planning/affordability/scenario outputs

---

# Communication Rules

Signal language should stay:
- calm
- plain-English
- non-judgemental
- realistic but slightly hopeful

Preferred style:
- acknowledge pressure directly
- explain what is driving it
- show what is still stable
- keep options visible

Avoid jargon-heavy phrases and avoid blame.
